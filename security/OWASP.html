Injection

Angriffszenario

SQL-Injection (dt. SQL-Einschleusung) bezeichnet das Ausnutzen einer Sicherheitslücke in Zusammenhang mit SQL-Datenbanken, die durch mangelnde Maskierung oder Überprüfung von Metazeichen in Benutzereingaben entsteht. Der Angreifer versucht dabei, über die Anwendung, die den Zugriff auf die Datenbank bereitstellt, eigene Datenbankbefehle einzuschleusen.

Gegenmassnahme

Eine wichtige Gegenmaßnahme besteht darin, Metazeichen in Benutzereingaben auszufiltern oder zu Maskieren (Escapen). Hierdurch kann die Gefahr von SQL-Injections gemildert oder sogar beseitigt werden.

Der sicherste Weg ist jedoch, die Daten vom SQL-Interpreter fernzuhalten. Dabei werden gebundene Parameter in Prepared Statements verwendet. Hierbei werden die Daten als Parameter an einen bereits kompilierten Befehl übergeben. Die Daten werden somit nicht interpretiert und eine SQL-Injection verhindert.

Fehler in Authentifizierung und Session-Management

Angriffszenario

Eine Session oder Anmeldedaten werden vom Angreifer abgefangen oder gebrutforced (da Passwörter nicht richtig gehashed + gesalted wurden) und somit hat der Angreifer Zugriff auf die jewiligen Userdaten.

Gegenmassnahme

Eine Multi-Faktor Authentifizierung implementieren. Default Credentials nicht verwenden wenn deployed wird. Für Usermanagement eine Überprüfung ob die Passwörter zu schwach sind implementieren. Captcha bei mehrfacher Fehlereingabe implementieren. Nach mehreren Fehlversuchen ein Timeout für den Login implementieren. Sicheren- Serverseitigen Sessionhandler implementieren und Session nicht per URL zurückgeben. Sessions schliessen nach Logout/ Timeout.

Verlust der Vertraulichkeit sensibler Daten

Angriffszenario

Über Man in de Middle Attacks werden sensitive Daten während ihrem transport abgegriffen werden. Dazu ist in den meisten Fällen ein manueller Angriff nötig.

Gegenmassnahme

Um Man in the Middle Attacks zu minimieren verhindern sollten so wenige sensitive Daten wie möglich übertragen werden und diese in jedem Fall verschlüsselt werden.


XML External Entities (XEE)

Angriffszenario

Ein Angreifer hat die Möglichkeit ein XML-File hochzuladen. Dabei ist das XML-File infiziert mit einem Befehl, welcher Datei vom Server auslesen möchte. Oder der Angreifer möchte durch das XML-File ein File auf dem Server ablegen.

Gegenmassnahme

Upload von XML nicht erlauben, dafür andere Datenformate wie JSON verwenden etc. Ein XML-Validator verwenden.
